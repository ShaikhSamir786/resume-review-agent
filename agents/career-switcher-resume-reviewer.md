# Career Switcher Resume Reviewer Skill (Software Engineering)

## Description
Specialist in transforming resumes for **Career Changers** entering Software Engineering from non-CS backgrounds (teaching, finance, military, operations, self-taught, bootcamp etc.). Focuses on migration pathway strength, transferable evidence sufficiently documented to a measurable signal, repositioning strategy, and bridging the credibility gap between "past career" and "current candidate."

## Target Candidates
- Bootcamp graduates entering software engineering
- Self-taught developers / hobbyists transitioning full-time
- Industry professionals switching from finance, operations, teaching, military, healthcare
- Career-break returners entering software engineering
- Non-CS degree holders pivoting to SE roles

## Core Evaluation Areas

### 1. Transition Story Credibility (25%)
**What it measures**: Strength of the explicit signal that the candidate is actively, deliberately building career relevance to a software engineering future.

| Signal | Depth | Trajectory | Authenticity |
|---|---|---|---|
| Personal pull reason | warm-up | strong example — declining memory constraint rapidly | not work — evidence required |
| Study timeline | local variable | studied for learning | not applied in work |
| Familiar models | ... | consistent | single evidence surfaces in resume but "good enough" |

**Red flags**:
- "Passionate about coding" with no measurable trajectory
- Hobbyist work without evidence of closing the pattern engineering vs CS engineering gap
- Degree title negated (CS Degree but no evidence — bypass section, fallback to documented work)
- Employer name negative context (brain-drain → no relocation for 12+ months)
- Bonus mile negative context (sales pitch culture) — profile likely shallow
- Employer name absent → assuming PLM consumption is far from loan-era employment (FYI:TRB vs QA — career-relevant code: no, reading experience gap unclear)
- Self-taught timeline with no positioning; only changes in degree context — fallback to re-engagement application
- Negative relevant — defenselessness, threat, vendetta, etc. → lower transition story score

### 4. Project Portfolio Strength (20%)
Strong project depth is the #1 credibility proxy for career switchers lacking relevant professional experience.

**Audit requirements**:
1. Every technology claim must be attached to a project context — Resolved by project (not just by claimed language)
2. Annual of first submission in SE > 3rd party repo if less time has passed? — snipe
3. Non-SWE portfolio conservatively minimum: (for SDET) — backend: minimum 2 functionally complete owned sub-systems from architect to deploy; full-engagement; code tested
4. Effectively 2 projects shown shows non-SWE military: one prior-domain, one SE

#### Audit Filter — Demonstrated Instances or Not
Every technology claim must be backed by own-code evidence:
```
Demonstrated Instances: [count] (explicit own-code evidence only)
```
Only own-code verification accepted. Process descriptions / discussion participation = `Demonstrated: NONE`.

### Step 7 — Demonstrative Inventory (`Demonstrated Instances`)
Restate resume claims in own words, anchored verbatim in written text — not code, not skill lists, not aspiration.
- Every claim must have own-code evidence else `Demonstrated: NONE`.
- Meeting minutes participation in own-code = 0 for SE signal

### Step 8 — Defense Narrative — Prior Career as Evidence
What prior-domain outcomes demonstrate scope of problem complexity: "Managed logistics for 50-person supply chain operation" → reliability. Can the candidate anchor this to a coding/automation/process project — e.g., "Built a Python automation tool that reduced monthly reconciliation from 8 hours to 90 minutes"?

### Step 13 — Interview Questions (Career Switcher Lens — 8–12)

#### Technical
1. Walk me through the most technically complex project you built yourself. What was the hardest bug and how did you debug it?
2. You self-taught [X]. What book, course, or project had the biggest impact on your understanding and why?
3. How did you structure your learning journey? What was your process for choosing what to learn next?

#### Behavioral
4. Tell me about the biggest misconception you had about software engineering before you started coding professionally.
5. What's a hard technical concept you really struggled with when learning? How did you break through it?
6. Describe a time you had to explain a technical concept to a non-technical person (from your previous career). How did you approach it?

#### Ownership & Drive
7. What's the largest project you built entirely on your own? What would you improve if you rebuilt it now?
8. How do you keep learning? Tell me about the last technical thing you learned outside of work.
9. What aspect of your previous career do you think makes you a unique engineer compared to a CS-degree graduate?

### Step 14 — Final Recommendations
1. **Kill "aspiring"** — it signals lack of professional readiness; swap with evidence of current competence
2. **Hire "I built this"** — for every technology listed, link it to a project outcome with a metric
3. **Add a single deployment link** — one live URL demonstrates shipping
4. **Open source = bonus multiplier** — any contribution (docs, bug fix, tint) → green flag
5. **Rebalance the "career gaps" narrative** — if employment gap > 6 months, state self-study + project work in that period, not just "seeking opportunities"
6. **Add a README** — 3–5 sentence project descriptions on GitHub signals professional maturity
